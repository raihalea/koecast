## §1 LLM 整形誤訂正の事例 ★最重要

`partial` / `final` (Riva 認識) は妥当だが、`formatted` (LLM 整形後) で意味が
変わってしまった事例。**これが 6-4-b の対策 (gateway 側 `llm.py` プロンプト
見直し + glossary 織り込み + 温度調整) の入力**になる。

既知事例 (段階6-3 で観測):
- 「ベッドロックで LLM を動かす」 → final「ベッドロックで ed を動かす。」 → formatted **「Bedrock EditionでEDを動かす。」** (LLM が「ベッドロック」を Minecraft Bedrock Edition と誤連想、`ed` を `ED` 大文字化)
- 「テスト注入」 → final「Tisto junior」 → formatted **「ティエス ト・ジュニア」** (Riva が誤認識 → LLM が固有名詞として整形)
- 「私は今マックで作業しています」 → final 同 → formatted **「私は今 Mac で作業しています」** (これは○。固有名詞英字化が望ましく効いた例)

## §5 overlay の表示挙動

枠なし最前面・フォーカス非奪取の overlay window の使い勝手。

観察ポイント:
- **フォーカス保持**: overlay が出てもアクティブアプリのキー入力が止まらないか
- **自動 hide のタイミング**: session_end 後 3 秒 で hide する仕様。早すぎ?
  遅すぎ? 連続発話のとき次のセッションで上書きされる挙動は適切か
- **表示位置・サイズ**: 画面左上 (100, 60) / 520×96。位置を変えたい? サイズ
  は読みやすいか
- **長文の partial / final の表示**: 一行で省略されている (`text-overflow:
  ellipsis`) が、これで十分か。複数行にする必要があるか
- **複数 segment の置換**: 同じセッションで 2 つめのセグメントが来たとき、前の
  segment が消える挙動の良し悪し

## §6 再接続まわり

WebSocket 切断 → 自動再接続の挙動。

観察ポイント:
- **gateway 一時停止**: DGX で `docker compose restart gateway` したときの
  Reconnecting → Ready 復帰の見え方 (6-3-b で確認済み、日常運用での再観察)
- **ネットワーク切断**: Tailscale OFF → 復活、Wi-Fi 切断 → 復活
- **長時間アイドル**: koecast 起動したまま 30 分以上放置 → ping/pong で接続
  維持されるか、いつのまにか切れていないか
- **接続できないとき**: server_url を不正な値にして、Reconnecting がぐるぐる
  回り続ける挙動 (Fatal にならないことを確認)
